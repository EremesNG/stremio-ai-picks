# Delta for Agent Contract

## ADDED Requirements

### Requirement: Forbid agent-emitted TMDB identifiers
The agent MUST emit recommendation items without `tmdb_id`. Any emitted `tmdb_id` field MUST be treated as a schema violation (`extra_field`) by schema validation.

#### Scenario: Agent emits legacy `tmdb_id`
- GIVEN the agent returns `[ { "type": "movie", "title": "Oxygen", "year": 2021, "tmdb_id": 18561 } ]`
- WHEN `validateAgentItems` evaluates the payload against `AGENT_ITEM_SCHEMA`
- THEN the item is rejected with an `extra_field` violation for `tmdb_id`
- AND the item is not schema-valid for collection

#### Scenario: Agent emits three-field contract
- GIVEN the agent returns `[ { "type": "series", "title": "Dark", "year": 2017 } ]`
- WHEN `validateAgentItems` evaluates the payload
- THEN the item is schema-valid
- AND no `tmdb_id` requirement is enforced on agent output

## MODIFIED Requirements

### Requirement: Keep schema wording identical across turn prompts
This change MODIFIES baseline requirement "Keep schema wording identical across turn prompts" (`openspec/specs/agent-contract/spec.md:5-13`). The schema wording used in `buildAgentSystemPrompt` and `buildTurnMessage` MUST be identical in field set and field names. Both prompts MUST refer to exactly three fields, spelled `type`, `title`, and `year`, and MUST NOT mention `tmdb_id`.

#### Scenario: Prompt helpers describe the same three fields
- GIVEN `buildAgentSystemPrompt` and `buildTurnMessage` are rendered for the same turn
- WHEN the agent reads both prompts
- THEN both prompts describe exactly `type`, `title`, and `year`
- AND neither prompt includes `tmdb_id`

### Requirement: Emit current-turn candidate arrays only
This change MODIFIES baseline requirement "Emit current-turn candidate arrays only" (`openspec/specs/agent-contract/spec.md:15-24`). The agent MUST return a single JSON array and nothing else for the current turn. Every array item MUST be an object with exactly three fields: `type` (string), `title` (string), and `year` (number). The agent MUST NOT include additional fields.

#### Scenario: Response includes non-contract field
- GIVEN a generation turn is active
- WHEN the agent emits an item containing `reason`, `overview`, `imdb_id`, or `tmdb_id`
- THEN the payload violates the contract
- AND corrective feedback MUST request the three-field schema only

### Requirement: Corrective feedback MUST reference the three-field schema
The corrective follow-up generated by `buildCorrectiveFeedback` MUST render the required schema from `AGENT_ITEM_SCHEMA` and MUST therefore reference only `type`, `title`, and `year`.

#### Scenario: Schema violations trigger corrected schema reminder
- GIVEN the first response in a turn violates schema
- WHEN orchestrator sends corrective feedback
- THEN the feedback includes the required output schema lines for `type`, `title`, and `year`
- AND the feedback does not instruct the agent to emit `tmdb_id`

### Requirement: Preserve favorites tool behavior and functionResponse round-trips
This change MODIFIES baseline requirement "Use favorites as optional selection context" (`openspec/specs/agent-contract/spec.md:48-55`).

`get_user_favorites` MUST remain agent-visible and MAY be called during a turn. Its functionResponse round-trip behavior MUST remain unchanged (tool call from model -> orchestrator tool execution -> functionResponse injected back into chat context).

#### Scenario: Favorites call continues to work in-turn
- GIVEN the agent calls `get_user_favorites`
- WHEN the orchestrator executes the tool and returns `functionResponse`
- THEN the agent can continue the same chat turn and emit a final JSON candidate array
- AND the final candidate array still follows the three-field output contract

## REMOVED Requirements

### Requirement: Resolve every candidate through TMDB search
This change REMOVES baseline requirement "Resolve every candidate through TMDB search" (`openspec/specs/agent-contract/spec.md:37-47`). TMDB resolution is no longer an agent responsibility.

#### Scenario: Agent tool surface excludes TMDB search
- GIVEN the orchestrator creates chat tools for the agent
- WHEN tool declarations are attached
- THEN `batch_search_tmdb` is absent from the agent-visible tool surface
- AND only `get_user_favorites` remains exposed
